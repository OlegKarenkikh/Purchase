# ТЕХНИЧЕСКОЕ ЗАДАНИЕ АИС УДЗ - ЧАСТЬ 2

## 4. НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

### 4.1. Требования к производительности

**NFR-1.1: Время отклика**
- Загрузка главной страницы: < 2 сек
- Загрузка документа (до 10 МБ): < 5 сек
- Анализ документации (до 50 стр): < 30 сек
- Формирование пакета документов: < 10 сек
- Генерация отчета: < 15 сек

**NFR-1.2: OCR производительность**
- Одна страница A4 (300 DPI): < 3 сек
- Документ 10 страниц: < 25 сек
- Точность распознавания: ≥ 95%

**NFR-1.3: Пропускная способность**
- 100 одновременных пользователей
- 500 запросов в минуту
- 50 параллельных анализов документации
- 1000 документов в день

### 4.2. Требования к надежности

**NFR-2.1: Доступность (Uptime)**
- Production: 99.5% (43.8 часа downtime в год)
- Плановые работы: по выходным с 00:00 до 06:00
- Время восстановления (MTTR): < 2 часа

**NFR-2.2: Резервное копирование**
- Полное: раз в неделю
- Инкрементное: каждый день
- Хранение: 30 дней
- Время восстановления из backup: < 4 часа

### 4.3. Требования к безопасности

**NFR-4.1: Аутентификация**
- JWT токены с refresh механизмом
- Lifetime: access token 15 мин, refresh token 7 дней
- 2FA для критичных операций (опционально)
- Блокировка после 5 неудачных попыток входа

**NFR-4.2: Авторизация (RBAC)**
- Администратор: полный доступ
- Менеджер закупок: создание и редактирование заявок
- Юрист: просмотр и экспертиза документов
- Бухгалтер: доступ к финансовым документам
- Читатель: только просмотр

**NFR-4.3: Шифрование**
- HTTPS (TLS 1.3) для всех соединений
- Шифрование паролей: bcrypt (cost 12)
- Шифрование чувствительных данных в БД: AES-256

**NFR-4.4: Защита от атак**
- SQL Injection: использование ORM
- XSS: санитизация ввода
- CSRF: токены
- Rate Limiting: 100 req/min на IP

### 4.4. Требования к интеграциям

**NFR-5.1: REST API**
- Следовать стандарту REST
- Использовать JSON для обмена данными
- Версионироваться (v1, v2)
- Документироваться через OpenAPI/Swagger

**NFR-5.2: Основные эндпоинты**
```
GET    /api/v1/procurements
POST   /api/v1/procurements
GET    /api/v1/procurements/{id}
PUT    /api/v1/procurements/{id}
DELETE /api/v1/procurements/{id}
POST   /api/v1/procurements/{id}/analyze
POST   /api/v1/procurements/{id}/generate-package
GET    /api/v1/documents
POST   /api/v1/documents/upload
GET    /api/v1/documents/{id}
```

### 4.5. Требования к пользовательскому интерфейсу

**NFR-6.1: Удобство использования**
- Интуитивно понятный интерфейс
- Минимальное обучение (< 2 часов)
- Поддержка горячих клавиш
- Адаптивный дизайн (responsive)

**NFR-6.2: Доступность (A11Y)**
- Соответствие WCAG 2.1 Level AA
- Поддержка screen readers
- Keyboard navigation
- Настраиваемый размер шрифта (14-20px)

### 4.6. Требования к совместимости

**NFR-7.1: Серверная часть**
- Linux: Ubuntu 22.04+, Debian 11+, RHEL 8+
- Docker: 24.0+
- Python: 3.10+
- PostgreSQL: 14+
- Redis: 7.0+

**NFR-7.2: Клиентская часть**
- Google Chrome: последние 2 версии
- Mozilla Firefox: последние 2 версии
- Microsoft Edge: последние 2 версии
- Safari: последние 2 версии (macOS)
- Минимальное разрешение: 1280x720

## 6. ПРИЕМКА И КРИТЕРИИ КАЧЕСТВА

### 6.1. Критерии приемки

**ACC-1:** Система принимается, если:

1. **Функциональность:** Реализовано ≥ 95% требований из раздела 3
2. **Производительность:** Соответствует требованиям из раздела 4.1
3. **Надежность:** Uptime ≥ 99% на тестовом окружении
4. **Тестирование:**
   - Code coverage ≥ 80%
   - Пройдены все критичные тесты
   - 0 Blocker багов
   - ≤ 5 Critical багов
5. **Документация:** Вся документация из п. 4.8 предоставлена
6. **Обучение:** Проведено обучение пользователей

### 6.2. Метрики качества

#### Метрики разработки
- **Code coverage:** ≥ 80%
- **Technical debt:** < 5% от общего кода
- **Bugs:**
  - Blocker: 0
  - Critical: ≤ 5
  - High: ≤ 20
  - Medium/Low: допустимы

#### Метрики производительности
- **Время отклика API:** p95 < 200ms, p99 < 500ms
- **Время анализа документа:** среднее < 25 сек
- **Uptime:** ≥ 99.5%

#### Метрики качества данных
- **Точность OCR:** ≥ 95%
- **Точность извлечения требований:** ≥ 90%
- **Полнота извлечения:** ≥ 95% (не пропускаем требования)

## 7. ГЛОССАРИЙ

### Термины и определения

**44-ФЗ** - Федеральный закон "О контрактной системе в сфере закупок товаров, работ, услуг для обеспечения государственных и муниципальных нужд"

**223-ФЗ** - Федеральный закон "О закупках товаров, работ, услуг отдельными видами юридических лиц"

**АИС УДЗ** - Автоматизированная информационная система управления документами закупочной деятельности

**ЕИС** - Единая информационная система в сфере закупок (zakupki.gov.ru)

**LLM** - Large Language Model, большая языковая модель

**OCR** - Optical Character Recognition, оптическое распознавание символов

**JWT** - JSON Web Token, токен для аутентификации

**RBAC** - Role-Based Access Control, управление доступом на основе ролей

**REST** - Representational State Transfer, архитектурный стиль для API

**SLA** - Service Level Agreement, соглашение об уровне сервиса

**Uptime** - Время доступности системы

**MTTR** - Mean Time To Repair, среднее время восстановления
