## 1. ОБЩИЕ СВЕДЕНИЯ

### 1.1. Полное наименование системы
**Автоматизированная информационная система управления документами закупочной деятельности** (сокращенное наименование: АИС УДЗ)

### 1.2. Назначение системы
Система предназначена для автоматизации процессов подготовки, формирования и контроля пакетов документов при участии в государственных и коммерческих закупках по 44-ФЗ, 223-ФЗ и другим нормативным актам.

### 1.3. Основания для разработки
- Потребность в повышении эффективности работы отдела закупок
- Необходимость минимизации рисков отклонения заявок из-за ошибок в документах
- Сокращение времени на подготовку документации на 70%
- Унификация процессов проверки документов
- Снижение количества отклоненных заявок на 80%

### 1.4. Целевая аудитория

#### Внутренние пользователи:
- **Менеджеры по закупкам** - основные пользователи, формирующие заявки
- **Юридический отдел** - юридическая экспертиза документов
- **Бухгалтерия** - проверка финансовых документов
- **Руководители отделов** - утверждение заявок
- **Администраторы системы** - управление настройками

#### Внешние интеграции:
- Единая информационная система закупок (ЕИС)
- Системы электронного документооборота
- Сервисы проверки контрагентов

## 2. ЦЕЛИ И ЗАДАЧИ СИСТЕМЫ

### 2.1. Основные цели

1. **Автоматизация анализа документации закупок**
   - Извлечение требований из различных форматов документов
   - Классификация требований по типам и категориям
   - Выявление явных и неявных требований

2. **Интеллектуальное формирование пакетов документов**
   - Автоматический подбор необходимых документов
   - Генерация недостающих документов из шаблонов
   - Валидация комплектности пакетов

3. **Многоэтапный контроль качества**
   - Автоматическая валидация формата и содержания
   - Юридическая экспертиза критичных документов
   - Финансовая проверка коммерческих предложений
   - Итоговый контроль перед отправкой

4. **Централизованное хранение**
   - Единая база документов компании
   - Реестр реквизитов и сертификатов
   - История участия в закупках

5. **Повышение точности**
   - Минимизация человеческих ошибок
   - Стандартизация процессов
   - Контроль сроков действия документов

### 2.2. Ожидаемые результаты

#### Количественные показатели:
- Сокращение времени подготовки заявки с 8-10 часов до 2-3 часов (70%)
- Снижение количества отклоненных заявок с 15-20% до 3-5% (80%)
- Увеличение количества обрабатываемых закупок на 150%
- Сокращение затрат на подготовку документов на 60%

#### Качественные показатели:
- Повышение прозрачности процессов
- Создание базы знаний о типовых требованиях
- Снижение зависимости от конкретных сотрудников
- Улучшение аудита и отчетности

### 2.3. Границы системы

#### Что входит в систему:
- Загрузка и парсинг закупочной документации
- Анализ требований с помощью LLM
- Формирование пакетов документов
- Многоэтапный контроль
- Хранилище документов
- API для интеграций
- Веб-интерфейс для пользователей

#### Что НЕ входит в систему:
- Электронная подпись документов (интеграция со сторонними КриптоПро)
- Бухгалтерский учет (интеграция с 1С)
- Мониторинг площадок закупок (интеграция с ЕИС)
- CRM функционал

## 3. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

### 3.1. Модуль загрузки и анализа закупочной документации

#### 3.1.1. Загрузка документов

**Требование FR-1.1:** Система должна поддерживать загрузку документов в следующих форматах:
- PDF (текстовые и отсканированные)
- DOCX, DOC
- RTF
- TXT
- ZIP-архивы с вложенными документами

**Требование FR-1.2:** Система должна поддерживать загрузку документов следующими способами:
- Drag & Drop в веб-интерфейсе
- Выбор файлов через диалог
- Загрузка через API
- Импорт из ЕИС по номеру закупки

**Требование FR-1.3:** Максимальный размер одного файла - 50 МБ

**Требование FR-1.4:** Максимальный размер ZIP-архива - 200 МБ

#### 3.1.2. Парсинг документов

**Требование FR-1.5:** Система должна автоматически извлекать текст из:
- Текстовых PDF
- Сканированных PDF с помощью OCR (точность >95%)
- DOCX с сохранением структуры
- Таблиц в PDF и DOCX

**Требование FR-1.6:** OCR должен поддерживать распознавание:
- Русского языка
- Английского языка
- Смешанных текстов
- Текста в таблицах
- Печатей и штампов

**Требование FR-1.7:** Система должна сохранять метаданные документов:
- Имя файла
- Размер
- Дата загрузки
- Автор (если доступно)
- Количество страниц

#### 3.1.3. Анализ требований с помощью LLM

**Требование FR-1.8:** Система должна использовать LLM для извлечения:
- Идентификационных данных закупки (номер, название, заказчик)
- Правовой основы (44-ФЗ, 223-ФЗ, коммерческая)
- Типа процедуры (аукцион, конкурс, запрос предложений)
- Перечня требуемых документов
- Требований к формату документов (оригинал/копия/электронная форма)
- Сроков действия документов
- Критериев оценки заявок

**Требование FR-1.9:** Система должна категоризировать документы по типам:
1. **Регистрационные** - выписка ЕГРЮЛ, устав, решения о создании
2. **Финансовые** - бухотчетность, справки из банка
3. **Специальные разрешительные** - лицензии, допуски СРО
4. **Кадровые** - штатное расписание, трудовые договоры
5. **Опыт и квалификация** - портфолио, рекомендации
6. **Техническая документация** - спецификации, чертежи
7. **Ценовое предложение** - коммерческое предложение, смета
8. **Обеспечительные** - банковская гарантия, договор страхования

**Требование FR-1.10:** Для каждого документа система должна определять:
- `id` - уникальный идентификатор
- `name` - название документа
- `category` - категория из списка выше
- `mandatory` - обязательность (true/false)
- `format` - требуемый формат (оригинал/копия/электронная)
- `validity_requirements` - требования к сроку действия
- `source_reference` - ссылка на раздел в документации
- `implicit` - был ли выявлен неявно (true/false)

**Требование FR-1.11:** Система должна выявлять неявные требования:
- Документы, необходимые по законодательству, но не указанные явно
- Документы, вытекающие из других требований
- Стандартные документы для данного типа закупки

#### 3.1.4. Вывод результата анализа

**Требование FR-1.12:** Система должна выводить результат в структурированном JSON формате:

```json
{
  "procurement_info": {
    "number": "0373200000123000001",
    "legal_basis": "44-ФЗ",
    "procedure_type": "Электронный аукцион",
    "customer": "ГКУ Управление капитального строительства"
  },
  "required_documents": [
    {
      "id": "doc_001",
      "name": "Выписка из ЕГРЮЛ",
      "category": "Регистрационные",
      "mandatory": true,
      "format": "Заверенная копия",
      "validity_requirements": "Не старее 30 дней",
      "source_reference": "Раздел III.3.4.1, п. 1",
      "implicit": false
    }
  ],
  "document_verification": {
    "provided": [],
    "missing_critical": [],
    "missing_optional": [],
    "issues": [],
    "completeness_score": 0
  },
  "critical_warnings": []
}
```

**Требование FR-1.13:** Система должна обеспечивать дедупликацию:
- Удаление полных дубликатов по точному совпадению названия
- Удаление похожих документов (схожесть >85%)
- Объединение документов с разными формулировками одного требования

**Требование FR-1.14:** Максимальное количество документов в одном анализе - 50

### 3.2. Модуль реестра документов

#### 3.2.1. Хранилище документов

**Требование FR-2.1:** Система должна вести централизованное хранилище документов компании:
- Регистрационные документы
- Финансовые документы
- Лицензии и сертификаты
- Портфолио и рекомендации
- Шаблоны документов

**Требование FR-2.2:** Для каждого документа должны храниться:
- Файл документа
- Название
- Тип/категория
- Дата выдачи
- Срок действия
- Орган, выдавший документ
- Номер документа
- Теги для поиска
- История изменений

**Требование FR-2.3:** Система должна отслеживать сроки действия документов:
- Уведомление за 30 дней до истечения
- Уведомление за 7 дней до истечения
- Пометка истекших документов
- Блокировка использования истекших документов

#### 3.2.2. Реестр реквизитов

**Требование FR-2.4:** Система должна хранить реквизиты компании:
- Полное наименование
- Сокращенное наименование
- ИНН, КПП, ОГРН
- Юридический адрес
- Фактический адрес
- Банковские реквизиты
- Контактные данные
- Данные руководителя

**Требование FR-2.5:** Система должна поддерживать версионность реквизитов:
- История изменений
- Дата начала действия
- Дата окончания действия
- Автоматический выбор актуальных реквизитов

#### 3.2.3. Поиск в реестре

**Требование FR-2.6:** Система должна обеспечивать поиск документов по:
- Названию
- Типу/категории
- Дате выдачи
- Сроку действия
- Тегам
- Полнотекстовому поиску по содержимому

**Требование FR-2.7:** Система должна поддерживать фильтрацию:
- По статусу (действующие/истекшие/истекающие)
- По категории
- По дате загрузки

### 3.3. Модуль формирования пакетов

#### 3.3.1. Автоматический подбор документов

**Требование FR-3.1:** На основе анализа требований система должна автоматически подбирать документы из реестра:
- Сопоставление требуемых документов с имеющимися
- Проверка актуальности документов
- Проверка соответствия формата требованиям

**Требование FR-3.2:** Система должна формировать чек-лист документов:
- Список всех требуемых документов
- Отметка о наличии/отсутствии
- Статус документа (актуален/истек/отсутствует)
- Приоритет (критичный/желательный)

**Требование FR-3.3:** Система должна рассчитывать полноту комплекта:
- Процент обеспеченности обязательными документами
- Процент обеспеченности всеми документами
- Оценка готовности к подаче заявки

#### 3.3.2. Генерация документов

**Требование FR-3.4:** Система должна генерировать недостающие документы из шаблонов:
- Анкета участника
- Декларации и гарантийные письма
- Доверенности
- Опись документов

**Требование FR-3.5:** Генерация должна включать:
- Подстановку реквизитов из реестра
- Заполнение данных о закупке
- Формирование документа в требуемом формате (DOCX, PDF)

#### 3.3.3. Формирование итогового пакета

**Требование FR-3.6:** Система должна формировать итоговый пакет документов:
- Сбор всех файлов в единую папку/архив
- Именование файлов согласно требованиям
- Генерация описи документов
- Подготовка к подаче в ЕИС или на площадку

**Требование FR-3.7:** Система должна поддерживать экспорт:
- ZIP-архив с документами
- Структурированная папка для локального хранения
- Прямая загрузка на электронную площадку (будущая функция)

### 3.4. Модуль многоэтапного контроля

#### 3.4.1. Этапы контроля

**Требование FR-4.1:** Система должна реализовывать 4 этапа контроля:

1. **Автоматический контроль**
   - Проверка форматов файлов
   - Проверка размеров файлов
   - Проверка читаемости документов
   - Проверка сроков действия
   - Проверка комплектности

2. **Юридический контроль**
   - Проверка соответствия требованиям закупки
   - Проверка актуальности нормативных ссылок
   - Проверка корректности формулировок
   - Выявление правовых рисков

3. **Финансовый контроль**
   - Проверка корректности расчетов
   - Проверка соответствия ценового предложения требованиям
   - Проверка налоговых документов

4. **Итоговый контроль**
   - Финальная проверка перед отправкой
   - Подтверждение руководителем
   - Регистрация факта подачи заявки

**Требование FR-4.2:** Для каждого этапа система должна формировать:
- Статус прохождения (passed/failed/warning)
- Список выявленных проблем
- Рекомендации по исправлению
- Ответственное лицо
- Дата и время проверки

#### 3.4.2. Правила валидации

**Требование FR-4.3:** Система должна проверять:
- Наличие всех обязательных документов
- Актуальность документов по срокам
- Соответствие форматов требованиям
- Читаемость отсканированных документов
- Наличие необходимых подписей и печатей
- Корректность реквизитов

**Требование FR-4.4:** Система должна классифицировать проблемы:
- **Blocker** - блокирующие, не позволяют подать заявку
- **Critical** - критичные, высокий риск отклонения
- **Warning** - предупреждения, желательно исправить
- **Info** - информационные сообщения

#### 3.4.3. Workflow контроля

**Требование FR-4.5:** Система должна поддерживать workflow:
- Последовательное прохождение этапов
- Возврат на предыдущий этап при выявлении проблем
- Параллельное прохождение независимых этапов
- Эскалация проблем руководству

**Требование FR-4.6:** Система должна отправлять уведомления:
- При назначении задачи на проверку
- При выявлении критичных проблем
- При успешном завершении этапа
- При истечении срока проверки

### 3.5. Модуль отчетности и аналитики

#### 3.5.1. Отчеты

**Требование FR-5.1:** Система должна формировать отчеты:
- По закупкам за период
- По отклоненным заявкам с причинами
- По срокам подготовки заявок
- По наиболее часто отсутствующим документам
- По загруженности сотрудников

**Требование FR-5.2:** Отчеты должны поддерживать:
- Экспорт в Excel
- Экспорт в PDF
- Построение графиков и диаграмм
- Фильтрацию по параметрам
- Группировку данных

#### 3.5.2. Аналитика

**Требование FR-5.3:** Система должна предоставлять аналитику:
- Динамика количества закупок
- Процент успешных заявок
- Средние сроки подготовки
- Наиболее востребованные документы
- Эффективность работы отделов
